The Reference

Package Cohesion

Open-Closed Principle

“Software entities should be open for extension, but closed for modification.”

Reuse-Release
Equivalence


“The granule of reuse is the granule of release.”


  • only release what you (re)use
  • keep your codebase clean

Common-Reuse
Principle

“Classes in a package are reused together. Reuse one and you reuse them all.”

Common-Closure
Principle


“The classes in a package should be closed together against the same kinds of changes. Changes affecting packges affect all classes in that package and no other packages.”

Package Coupling

Acyclic-Dependency
Principle


“Allow no cycles in the package-dependency graph.”

Stable-Dependency
Principle


“Depend in the direction of stability! ... Stable is what is not easily moved.”


Stability Metrics

Afferent Couplings $C_{a}$
Number of classes outside this package that depend on it

Efferent Couplings $C_{e}$
Number of classes inside this package that depend on outside classes

Instability $I$
$$ I = \frac{C_{e}}{C_{e} + C_{a}} $$

Stable-Abstractions
Principle


“A package should be as abstract as it is stable.”

Number of classes inside package $N_{c}$
Number of abstract classes inside package $N_{a}$

Abstractness $A$
$$ A = \frac{N_{a}}{N_{e}}$$

Summary

  • Modular reusable software contains dependencies
  • Dependencies should be clean
  • Dependencies should be directed
  • Stability and abstractness should be balanced
  • Metrics are good, but may be misleading
